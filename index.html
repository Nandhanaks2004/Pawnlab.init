PHASE 01

Network Discovery

Before attacking anything, we have to find it. Discovery is the very first trust boundary an attacker crosses: from "outside the network" to "aware of a live host."

Attacker's Objective

Identify which IP address on the local subnet belongs to the target machine, since a fresh VM rarely advertises itself.

🧠 Why this step exists
+

Every subsequent phase needs a target IP. On an isolated lab network there's no DNS entry to look up, so attackers fall back to Layer 2 discovery — asking "who has this hardware address?" across the whole subnet and recording who answers.

⚙ How ARP works internally
+

ARP (Address Resolution Protocol) maps IP addresses to MAC addresses. netdiscover broadcasts "who has 192.168.1.X?" to every host on the segment. Any live host replies with its MAC address, revealing its presence even if no other service is listening.

Because ARP operates below IP routing, this technique only works on the local broadcast domain — it will not discover hosts across routers.

👁 What to Observe
+

Look at the vendor OUI portion of the MAC address — it often reveals whether the host is a VirtualBox/VMware VM, which helps confirm you're looking at the intended lab target rather than your own router or another device.

⚠ Common Beginner Mistakes

Running netdiscover on the wrong network interface, or forgetting that a freshly booted VM may take a few seconds before it starts responding to ARP requests.

kali@kali: ~

Q. Why can't netdiscover find hosts on a different subnet across a router?
Because routers block all ICMP traffic by default
Because ARP operates at Layer 2 and does not cross routing boundaries
Because netdiscover requires a static IP to function
ARP resolves addresses only within the local broadcast domain. Routers form the boundary of that domain, so ARP broadcasts never cross them.

PHASE 04

Local File Inclusion (LFI) Discovery

This is the pivotal vulnerability of the whole box. We go deep here: what LFI is, how it differs from related bugs, and how it actually executes inside the PHP engine.

LFI

The application includes a file from the local filesystem based on user input, without properly validating the path.

RFI

The application includes a file from a remote URL supplied by the attacker — requires allow_url_include to be enabled.

Directory Traversal

A broader bug class: using ../ sequences to escape an intended directory. LFI is often exploited using directory traversal.

How include() Works Internally
User Input (?page=...)
↓
include($_GET['page'] . '.php')
↓
PHP Engine resolves the path
↓
Reads file from disk
↓
Displays / Executes Result
🧠 What is Local File Inclusion
+

PHP's include() and require() statements load and execute another file at the point they're called. If the filename comes from user input and isn't restricted to a safe allow-list, an attacker can point it anywhere the web server process can read.

👁 What to Observe
+

Error messages that leak the server's include path (e.g. failed to open stream) confirm the parameter reaches a file-inclusion function, even before you get a working payload.

⚠ Common Beginner Mistakes

Assuming LFI failed just because ../../etc/passwd didn't work. Many apps append a fixed extension like .php, which silently breaks naive traversal payloads — that's exactly what happens here.

http://192.168.1.8/?page=home
Welcome to PawnLab — please log in to continue.
Why this happens

This is the normal, intended case: page=home resolves to home.php, a file that legitimately exists in the app directory.

Q. Why does ?page=../../etc/passwd fail here even though the app is vulnerable to LFI?
Because Apache blocks all traversal sequences by default
Because the app appends a fixed extension, turning the request into /etc/passwd.php, which doesn't exist
Because /etc/passwd is owned by root and unreadable by www-data
Many vulnerable includes look like include($_GET['page'] . '.php'). Appending .php to a traversal payload usually breaks it, which is why we need php://filter instead.
Q. What's the key difference between LFI and RFI?
RFI is always more severe than LFI
RFI includes a file from a remote, attacker-controlled URL; LFI includes a file already on the local disk
LFI only works over HTTPS
Both abuse the same include() sink — the difference is where the included file physically lives, and RFI additionally requires allow_url_include to be enabled in php.ini.
PHASE 05

PHP Filter Chain — Reading Source Without Executing It

Since .php gets appended automatically, we can't traverse to arbitrary files — but we can abuse a PHP stream wrapper to read the application's own source code.

php://filter

A built-in PHP stream wrapper that applies a filter to data as it's read — originally meant for things like on-the-fly compression, not for attackers. But include() doesn't know the difference between a filename and a stream wrapper URL.

convert.base64-encode

This specific filter Base64-encodes whatever it reads before handing it back to include(). Because the output is now Base64 text, PHP never recognizes it as executable <?php ?> code — it just gets echoed to the page as harmless-looking text.

Why source code is normally hidden

PHP files execute server-side — a browser never normally sees the raw <?php ... ?> source, only its output. This filter is what lets us "peek behind the curtain."

PHASE 08

File Upload Vulnerability

This is the single most consequential chapter in the walkthrough — the point where information disclosure turns into code execution.

How Uploads Normally Work
  • Client-side validation — JavaScript checks the file extension before submitting. Cosmetic only; trivially bypassed by editing the request.
  • Server-side extension checks — reject filenames ending in .php, .phtml, etc.
  • MIME type checks — inspect the Content-Type header sent by the browser.
  • Magic byte / file signature checks — inspect the first bytes of the file content itself, since headers and extensions are both attacker-controlled.
Why Upload Vulnerabilities Happen

Most real-world upload bugs come from validating only one of these layers — commonly just the extension or just the MIME type — while trusting the rest.

A famous real-world pattern: an app blocks .php but allows .php5, .phtml, or double extensions like .php.jpg, because the deny-list was incomplete rather than an allow-list.

Why shell.php fails

The extension deny-list explicitly blocks .php, so the upload is rejected before the file ever touches disk — regardless of what's inside it.

Why shell.php.gif works

The final extension is .gif, an allowed image type. If a magic-byte check exists too, prepending the literal bytes GIF87a satisfies it — the file "looks like" a GIF to both checks while still containing executable PHP.

🧬 The GIF87a Magic Header

GIF87a is the literal ASCII signature that real GIF files begin with. Signature checks typically just confirm the first few bytes match a known format — they don't parse the entire file, so anything can follow those bytes, including a PHP tag.

PHASE 09

Cookie Exploitation — Triggering the Upload via LFI

The uploaded file is inert until something on the server includes it. This app makes that trivial through a cookie named lang.

What the lang Cookie Does

The application was designed to let users pick a language, e.g. lang=en, which the server turns into include('lang/en.php') to load translated text. The developer trusted this cookie value completely.

Why ../upload/file.gif Executes

Because the cookie value is concatenated directly into an include() path, setting it to a traversal string pointing at our uploaded file makes the server include our file instead of a language file. PHP executes anything between <?php tags regardless of the file's extension.

PHASE 10

Reverse Shell

One-off commands through a URL are clumsy. We upgrade to a full interactive shell by making the target connect back to us.

Reverse vs Bind Shell
Reverse Shell

The target initiates the connection back to the attacker. Works even when the target sits behind a firewall that blocks inbound connections, since outbound traffic is usually less restricted.

Bind Shell

The target opens a listening port and waits for the attacker to connect inbound. Simpler, but frequently blocked by firewalls or NAT.

PHASE 11

Shell Stabilization

A raw netcat shell is fragile: no tab-completion, no job control, and Ctrl+C kills it outright. We upgrade it to a full interactive TTY.

Before — Broken Shell

No command history, no arrow keys, su fails silently, text editors render garbled — because there is no real pseudo-terminal (PTY) attached.

After — Interactive Shell

Full line editing, working su/sudo prompts, correctly sized terminal, functional text editors — a shell that behaves like a normal SSH session.

stabilization sequence
# 1) spawn a PTY on the target python -c 'import pty; pty.spawn("/bin/bash")' # 2) background the shell — press: CTRL+Z # 3) fix local terminal settings on Kali stty raw -echo # 4) bring the shell back to the foreground fg (press Enter once) # 5) set a proper terminal type export TERM=xterm reset (optional) # 6) verify the interactive TTY tty /dev/pts/0
pty.spawn()

Uses Python's pty module to allocate a real pseudo-terminal on the target and attach /bin/bash to it.

stty raw -echo

Tells your local terminal to stop processing control characters itself, passing them through to the remote PTY instead.

fg

Resumes the backgrounded remote shell so it once again receives your keystrokes directly.

TERM=xterm

Sets terminal capabilities so full-screen programs (editors, less, top) render correctly.

PHASE 12

Privilege Escalation — Concepts

Before hijacking anything, we need the vocabulary: how Linux tracks identity and how it decides which program runs when you type a bare command name.

UID

The real user ID — who actually launched the process. This is what commands like whoami reflect in most cases.

EUID

The effective user ID — whose permissions the kernel actually checks for file access. Normally equals the UID, except when SUID changes it.

SUID

A special file permission bit. When set on an executable, running it sets the process's EUID to the file's owner, not the user who ran it.

The PATH Variable

When you type cat without a full path, the shell doesn't magically know where cat lives — it searches each directory listed in the PATH environment variable, in order, and runs the first matching executable it finds.

Why this is dangerous

If an attacker-writable directory (like /tmp) appears earlier in PATH than the real system directories, and a privileged program calls a command like cat without its absolute path, the attacker's fake binary runs instead — inheriting whatever privileges that program had.

PHASE 13

SUID Exploitation → Root

The final hop. mike owns access to a SUID binary that mishandles user-supplied input.

mike@pawnlab: /home/mike
ls -la -rwsr-sr-x 1 root root msg2root # note the "s" — SUID + SGID, owned by root ./msg2root Message for root: hello && /bin/sh # # root shell spawned id uid=1002(mike) euid=0(root) /bin/cat /root/flag.txt .-=~=-. If you are reading this, you have broken 'init' PawnLab.
Why hello && /bin/sh Works

The msg2root binary passes our answer to a shell without sanitizing it, most likely via something like system() or popen(). The && operator is shell syntax, not part of any "message" — it chains a second command that only runs if the first succeeds. Because this all executes inside a shell spawned by the SUID process, our injected /bin/sh inherits the process's effective UID: root.

🔓 This is Command Injection

Untrusted input reaching a shell interpreter without sanitization or use of a safe exec-family call (which doesn't invoke a shell at all) is the root cause. Combined with the SUID bit, it becomes a direct path to root.

🔑 Why /bin/cat (absolute path) Matters Here

We explicitly call /bin/cat rather than bare cat — after all our PATH manipulation earlier, we want to guarantee we're running the real system binary to read the flag, not any leftover fake one. The kernel checks EUID at the moment of file access, and since our EUID is now 0, reading a root-only file succeeds.

Detection Perspective

Every step an attacker takes leaves traces somewhere. Here's what a defender monitoring this box would see.

Apache / Access Logs

Repeated requests with php://filter or ../ sequences in the page parameter; a spike in 404s from directory brute-forcing.

PHP / Application Logs

Warnings like failed to open stream, or a file being included from an upload/ path — never a legitimate application flow.

Authentication Logs

A successful login for kent immediately after (or correlated with) unusual database access patterns.

File Upload Logs

A .gif upload that is subsequently requested with a cmd= query parameter — GIFs should never receive query-string command execution.

Reverse Shell Indicators

Apache's child process (www-data) spawning /bin/bash with a network socket as stdin/stdout — a strong EDR signal.

Privilege Escalation

A world-writable file named cat appearing in /tmp; PATH environment variable modified in a shell session; unexpected SUID-binary execution just before a UID/EUID mismatch appears in audit logs.

Blue Team

Mitigation — Fixing Every Link in the Chain

LFI / include()

Never pass user input directly to include()/require(). Use a hard-coded allow-list mapping short keys to real filenames, and disable allow_url_include.

Secrets in Source

Store credentials outside the web root, in environment variables or a secrets manager — never in a PHP file a stream-wrapper trick can expose.

Password Storage

Use a slow, salted hashing algorithm (bcrypt, argon2) — never Base64 or a fast unsalted hash — for anything resembling a credential.

File Upload

Validate with an allow-list of extensions AND re-encode/re-generate the file server-side (e.g. resize the image), which strips any embedded payload. Store uploads outside the web root or serve them with a Content-Disposition that prevents execution.

Cookie / Session Trust

Treat cookies as fully attacker-controlled input — never build filesystem paths from them without strict validation.

PATH Hijacking

Privileged scripts and binaries should always call system utilities via their absolute path (/bin/cat, not cat), and should set a known-safe PATH explicitly at startup.

SUID Binaries

Avoid SUID entirely where possible; where required, drop privileges as early as possible and never pass user input to a shell interpreter.

Command Injection

Use exec-family calls with argument arrays instead of shell-invoking functions like system()/popen(), and never concatenate user input into a shell command string.
